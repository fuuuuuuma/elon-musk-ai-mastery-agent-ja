# 🎁 GrokBuildで使えるSkills 30選

Grok Build は Claude Code の Skill（`.claude/skills/<name>/SKILL.md`）・設定・MCP をそのまま取り込めるエージェント型コーディングCLIです。つまりClaude Code用に書いたSkillがそのままGrok Buildでも動きます。ここではどちらでも使える実用Skillを30個まとめました。

**受け取り方**: 下のコードブロック右上のコピーボタンでコピーするか、このファイルをダウンロードしてください。

**使い方**: 各Skillの`text`ブロックを`.claude/skills/<skill-name>/SKILL.md`として保存してください。Grok Build / Claude Codeはフォルダ内のSKILL.mdを自動で読み込み、descriptionに合う場面で自分から発火します。

## 開発・実装

### 1. plan-before-code（着手前プラン確認）
description: 複数ファイルにまたがる実装や仕様が曖昧なタスクを受けたとき、コードを書き始める前に使う。

```text
1. 要求を読んだら、まずPlan Modeで方針をまとめる（変更対象ファイル・手順・懸念点）。
2. 方針をA案/B案のように2〜3パターンで提示し、ユーザーの承認を待つ。
3. 承認が出るまで実際の編集・実行コマンドは行わない。
4. 承認後にプラン通り実装し、完了時に「変えた箇所」を一覧で報告する。
```

### 2. parallel-refactor（並列リファクタ）
description: 影響範囲が広いリファクタ（命名変更・API移行・型定義の一括更新など）を複数ファイルに適用するとき使う。

```text
1. 変更対象ファイルを機能単位でグループ分けする。
2. グループごとにgit worktreeを切り、最大8並列までsubagentに割り当てる。
3. 各subagentには「変更対象ファイルのパス」「変更ルール」「触ってよい範囲」を明記して渡す。
4. 全subagent完了後、差分をまとめて1本のブランチにマージし、テストを通す。
5. worktreeはマージ確認後に削除する。
```

### 3. bug-repro-first（再現優先）
description: バグ報告や「動かない」という指摘を受けたとき、修正コードを書く前に使う。

```text
1. 症状ではなく再現条件（入力・環境・手順）を特定するまで修正案を出さない。
2. 再現できたら最小構成のテストケースを1つ作る。
3. 原因を仮説→検証のサイクルで絞り込み、仮説を言語化してから直す。
4. 修正後は同じ再現手順で直ったことを確認してから完了報告する。
```

### 4. worktree-experiment（試し撃ちworktree）
description: 本線に影響させたくない大きな変更や検証的な実装を試すとき使う。

```text
1. 現在のブランチから新しいgit worktreeを作成する。
2. worktree内で自由に変更・実行・失敗を繰り返す。
3. 方向性が固まったら本線ブランチに差分を移植する。
4. 不採用になったworktreeはブランチごと削除して片付ける。
```

### 5. dependency-audit（依存追加前チェック）
description: 新しいライブラリやMCP・パッケージを追加する提案が出たとき使う。

```text
1. 追加候補のライセンス・最終更新日・メンテ状況を確認する。
2. 既存依存で同じ機能をまかなえないか先に確認する。
3. 追加理由と代替案を1〜2行で並記してユーザーに提示する。
4. 承認後にインストールし、ロックファイルの差分だけをコミットする。
```

## 情報収集・リサーチ

### 6. source-triangulate（複数情報源の裏取り）
description: 日付・数値・固有名詞など事実に関わる主張を書く前に使う。

```text
1. 一次情報源（公式サイト・公式ドキュメント・本人発信）を最優先で探す。
2. 一次情報が見つからない場合は「未確認」と明示し、断定しない。
3. 複数の情報源が食い違う場合は両方を併記し、判断を保留する。
4. 出典が確認できた項目のみ本文に反映する。
```

### 7. competitor-scan（競合比較調査）
description: 類似サービス・競合ツールを横並びで比較したいとき使う。

```text
1. 比較対象を3〜6個に絞り、subagentを対象数だけ並列起動する。
2. 各subagentには「機能」「価格」「対象ユーザー」など同じ項目セットで調べさせる。
3. 結果を1つの比較表にまとめ、出典が取れない項目は空欄にする。
4. 主観的な優劣コメントは最後に別枠でまとめる。
```

### 8. changelog-watch（変更点の要約）
description: 依存しているツール・ライブラリ・サービスの更新情報を確認したいとき使う。

```text
1. 公式のリリースノート/changelogページを探す。
2. 自分のプロジェクトに影響する変更点だけを抽出する。
3. 破壊的変更（Breaking Change）には印をつけて別枠で目立たせる。
4. 対応要否と対応期限の目安を末尾に添える。
```

### 9. spec-diff-checker（仕様と実装の差分確認）
description: 仕様書・設計メモと実際のコードが一致しているか確認したいとき使う。

```text
1. 仕様書側の要件を箇条書きで洗い出す。
2. 実装側を読み、各要件に対応する箇所があるか1つずつ照合する。
3. 未実装・仕様と異なる挙動をリスト化する。
4. 意図的な仕様変更なのか実装漏れなのか、判断がつかない項目は質問として残す。
```

### 10. research-brief（調査スコープ確定）
description: 調査依頼が「〇〇について調べて」のように範囲が曖昧なとき、着手前に使う。

```text
1. 何のための調査か（目的）と、成果物の形式を先に確認する。
2. 範囲が広い場合は2〜3個のサブテーマに分割する。
3. サブテーマごとに担当subagentを割り当て、並列で調査させる。
4. 各subagentの結果を1つのレポートに統合し、矛盾点は明示する。
```

## ドキュメント・発信

### 11. readme-from-code（コードからREADME生成）
description: 実装が固まったプロジェクトのREADMEが古い、または存在しないとき使う。

```text
1. package.json/エントリーポイント/主要ディレクトリ構成を読み取る。
2. セットアップ手順・使い方・主要コマンドを実際に動かして確認してから書く。
3. 「動きます」と書く前に、書いた手順を実際に再実行して検証する。
4. 既存READMEがある場合は全面書き換えせず、変わった節だけ更新する。
```

### 12. changelog-writer（コミット履歴からCHANGELOG生成）
description: リリース前にCHANGELOGを更新する必要があるとき使う。

```text
1. 前回リリースからのコミットログを取得する。
2. 機能追加・修正・破壊的変更のカテゴリに仕分ける。
3. ユーザー影響のある変更だけを日本語1行で要約する。
4. バージョン番号と日付を明記して既存CHANGELOGの先頭に追記する。
```

### 13. sns-post-drafter（SNS投稿文ドラフト）
description: 完成した動画・記事・リリースからX/note向けの告知文を作りたいとき使う。

```text
1. 成果物の核（誰に何が嬉しいか）を1文で抽出する。
2. 口語のキャッチ→説明2〜3段落→URLの順で組み立てる。
3. 太字記法や誇張表現は使わず、事実ベースで書く。
4. 公開前にファクトチェック工程を通す。
```

### 14. explainer-doc（非エンジニア向け解説）
description: 開発者以外に向けて機能や仕組みを説明する資料を作るとき使う。

```text
1. 専門用語は日常語・比喩に置き換える（例:「並列処理」→「同時に何人かで手分けする」）。
2. 「困りごと→こう頼むと解決する」の対応形式で例を作る。
3. 難易度の高い例は本文と分けて「発展編」として別枠に置く。
4. 読み終えて何をすればいいかが分かる一言で締める。
```

### 15. slide-outline（スライド構成案作成）
description: 資料や動画の「構成」だけを渡されて、スライドの流れを組み立てたいとき使う。

```text
1. 渡された構成を章立てに分解し、各章の要点を1行に圧縮する。
2. 章ごとに「結論→根拠→具体例」の順で並べる。
3. 図・表・比較が要る箇所を洗い出してメモを残す。
4. 完成した章立てをユーザーに見せてから本文作成に進む。
```

## 自動化・運用

### 16. goal-loop-runner（ゴール自律実行）
description: 明確なゴールがあり、達成するまで反復的に作業を進めたいとき使う。

```text
1. ゴールと完了条件を1文で定義する（例:「テストが全部通ったら終了」）。
2. ターン上限を決め、上限に達するかゴール達成で自動停止する設計にする。
3. 各反復の終わりに現在の進捗と次の一手を短く記録する。
4. 停止条件を満たしたら結果と未達項目を分けて報告する。
```

### 17. recurring-check（定期監視タスク）
description: 一定間隔で状態確認したい作業（デプロイ状況・投稿予約・在庫チェックなど）があるとき使う。

```text
1. 監視対象と確認間隔（例: 5分ごと）を決める。
2. 1回分のチェック内容を短いコマンド/プロンプトに落とし込む。
3. 状態に変化があったときだけ報告し、変化がなければ静かに継続する。
4. 想定外のエラーが連続したら自動で止めて人に知らせる。
```

### 18. cleanup-sweep（作業環境の掃除）
description: 未使用ブランチ・古いworktree・中間ファイルが溜まってきたとき使う。

```text
1. マージ済みブランチ、使われていないworktree、期限切れの一時設定を洗い出す。
2. まずdry-runで削除候補一覧だけを提示する。
3. ユーザー確認後に削除を実行する。
4. 中間ファイル（.dump.txt等）は都度削除し、キャッシュ系は残す。
```

### 19. hook-gate（危険操作の確認フック）
description: 削除・上書き・force push・外部送信など取り消せない操作を伴う作業を設計するとき使う。

```text
1. 取り消せない操作の一覧を洗い出す(削除/上書き/公開/課金/自動投稿など)。
2. 各操作の直前に確認ステップを挟む設計にする。
3. 確認なしで実行してよい範囲(dry-run・読み取り専用)を明確に区別する。
4. 確認ステップをスキップする条件は作らない。
```

### 20. deploy-checklist（デプロイ前チェック）
description: 本番反映・公開作業の直前に使う。

```text
1. テスト・ビルド・lintが通っていることを確認する。
2. 環境変数やシークレットがコミットに含まれていないか確認する。
3. ロールバック手順を1行で書き出しておく。
4. 実際にデプロイ先の環境で動作確認するまで「完了」と言わない。
```

## データ整形

### 21. csv-normalize（表記ゆれ正規化）
description: 複数の担当者・時期で作られたCSVやスプレッドシートを1つの形式にそろえたいとき使う。

```text
1. 列名・日付形式・全角半角などの表記ゆれを洗い出す。
2. 正規化ルール(統一する側の表記)を先に決めてから一括変換する。
3. 変換後、件数・合計値など件数系の数字が変わっていないか検算する。
4. 変換前後の差分をサンプル数行で提示して確認を取る。
```

### 22. log-triage（ログの分類・異常検知）
description: 大量のログやエラー出力から異常だけを拾い出したいとき使う。

```text
1. ログをエラー/警告/情報のレベルでまず分ける。
2. 頻出パターンをグルーピングし、件数の多い順に並べる。
3. 初見のエラーパターンや急増しているものを優先して報告する。
4. 既知で無害と分かっているものは折りたたんで本文を短くする。
```

### 23. schema-migrate（データスキーマ移行）
description: データ構造（カラム追加・型変更・テーブル分割など）を変更するとき使う。

```text
1. 変更前後のスキーマ差分を明文化する。
2. 既存データを移行するスクリプトを書き、まず少数サンプルで試す。
3. 本番相当データに適用する前に、件数・欠損値をチェックする。
4. 移行後はロールバック用のバックアップを取ってから本適用する。
```

### 24. json-to-markdown-table（構造化データの表整形）
description: API応答やJSONログを人が読める表に変換したいとき使う。

```text
1. JSONのキーを列名候補として洗い出す。
2. 見せたい項目だけを選び、余分なメタ情報は列から外す。
3. Markdown表に変換し、値が長い列は要約するか折り返す。
4. 元データへのアクセス方法(ファイルパスやクエリ)を表の下に添える。
```

### 25. duplicate-finder（重複データ検出）
description: 同じ内容が複数箇所に散らばっている疑いがあるとき（顧客リスト・タグ・ファイルなど）使う。

```text
1. 完全一致だけでなく表記ゆれ(全角半角・スペース・大文字小文字)も候補に含める。
2. 重複候補をペアで提示し、どちらを正とするか機械的に決めつけない。
3. 統合ルール(新しい方を残す、情報量が多い方を残す等)を先に確認する。
4. 統合後は件数の変化を報告する。
```

## 品質・レビュー・セーフティ

### 26. multi-agent-review（多視点並列レビュー）
description: 完成したコードや文章を提出・公開する前に、複数の観点でチェックしたいとき使う。

```text
1. レビュー観点(バグ/簡潔化/セキュリティ/事実確認など)を3〜4個に分ける。
2. 観点ごとにsubagentを並列で起動し、それぞれ1つの観点だけに集中させる。
3. 各subagentの指摘を1つのリストに統合し、重複を1本化する。
4. 指摘への対応要否はユーザーに委ね、自動で書き換えない。
```

### 27. fact-check-gate（公開前ファクトチェック）
description: 外部に出る文章(SNS投稿・note記事・概要欄など)を公開する直前に使う。

```text
1. 文中の日付・数値・固有名詞・引用を1つずつ洗い出す。
2. それぞれ出典に当たれるか確認し、当たれない項目は「未確認」と明記する。
3. 煽り表現(「絶対」「衝撃」のような断定・扇情語)がないか通しで確認する。
4. 修正が必要な箇所だけをユーザーに提示し、公開判断は委ねる。
```

### 28. secrets-guard（機密情報の流出防止）
description: コミット・共有・公開の直前に、鍵やトークンが混入していないか確認したいとき使う。

```text
1. コミット対象の差分にAPIキー・トークン・パスワードらしき文字列がないか走査する。
2. .env系ファイルがgitignoreされているか確認する。
3. 検出した場合はコミットを止め、該当箇所をユーザーに報告する。
4. 誤って履歴に残った場合はローテーション(キー失効・再発行)が要ることを伝える。
```

### 29. claude-md-sync（運用ルールの参照・反映）
description: セッション開始時や、同じ指摘を2回以上受けたときに使う。

```text
1. 作業開始前にCLAUDE.md/AGENTS.mdを読み、プロジェクト固有ルールを把握する。
2. ルールと矛盾する提案をしそうなときは、着手前に確認を取る。
3. 同じ修正指示を2回以上受けた項目は、該当ファイルに追記する形で残す。
4. ルールファイルは重複を避け、該当する階層(グローバル/プロジェクト/配下)にだけ書く。
```

### 30. scope-lock（編集範囲の固定）
description: 指示された箇所だけを直せばよい編集・修正タスクを受けたとき使う。

```text
1. 指示された段落・関数・ファイルの範囲を最初に明確にする。
2. 範囲外に「ついで改善」したい箇所を見つけても、その場では触らない。
3. 改善案は末尾に別枠のメモとして添えるだけにとどめる。
4. 完了時に「変えた箇所/触っていない箇所/確認してほしい箇所」を分けて報告する。
```
